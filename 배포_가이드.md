# SOOSAN 입찰 · 견적 시스템 – GitHub + Replit 배포 가이드

## 흐름 요약

1. **로컬(Cursor)** 에서 코드 수정 후 **GitHub** 로 푸시  
2. **Replit Shell** 에서 GitHub 최신 코드로 동기화  
3. **Replit** 에서 Republish 로 재배포  

---

## 사전 준비 (최초 1회)

### Supabase DB 마이그레이션

배포 전 Supabase에 입찰 시스템 테이블이 있어야 합니다.

1. [Supabase Dashboard](https://supabase.com/dashboard) → 해당 프로젝트 → **SQL Editor**
2. 저장소의 `supabase/apply-all-migrations.sql` 내용을 붙여넣기 → **Run**
3. **Table Editor**에서 아래 테이블 생성 확인  
   - `crawl_sites`, `khnp_bid_notice`, `khnp_bid_open`, `khnp_bid_private`, `khnp_bid_plan_spec`
   - `user_bid_favorites`, `user_bid_notice_memos`, `user_bid_submissions`, `user_estimate_submissions`, `user_order_reports`
   - `bid_notice_attachments`, `user_bid_notice_screening`, `user_order_report_summaries`
   - `departments`, `bid_notice_assignments`, `bid_notice_screening_keywords`, `bid_users`
4. **Storage** 버킷 확인: `bid-notice-attachments`, `order-report-summaries`
5. **Project Settings → API → Reload schema** (또는 1~2분 대기)

> 이미 마이그레이션을 적용한 DB라면, `apply-all-migrations.sql`의 `IF NOT EXISTS` / `ON CONFLICT` 구문 덕분에 중복 실행해도 대부분 안전합니다.  
> KPOS(`khnp_bid_*`) 테이블이 이미 있으면 001 CREATE 구문은 자동으로 건너뜁니다.

### RLS 오류 예방 (권장)

데이터 저장 시 `new row violates row-level security policy` 오류가 나면 SQL Editor에서 한 번 실행:

```sql
ALTER TABLE crawl_sites DISABLE ROW LEVEL SECURITY;
ALTER TABLE khnp_bid_notice DISABLE ROW LEVEL SECURITY;
ALTER TABLE khnp_bid_open DISABLE ROW LEVEL SECURITY;
ALTER TABLE khnp_bid_private DISABLE ROW LEVEL SECURITY;
ALTER TABLE khnp_bid_plan_spec DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_bid_favorites DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_bid_notice_memos DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_bid_submissions DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_estimate_submissions DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_order_reports DISABLE ROW LEVEL SECURITY;
ALTER TABLE bid_notice_attachments DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_bid_notice_screening DISABLE ROW LEVEL SECURITY;
ALTER TABLE user_order_report_summaries DISABLE ROW LEVEL SECURITY;
ALTER TABLE bid_notice_screening_keywords DISABLE ROW LEVEL SECURITY;
ALTER TABLE departments DISABLE ROW LEVEL SECURITY;
ALTER TABLE bid_notice_assignments DISABLE ROW LEVEL SECURITY;
ALTER TABLE bid_users DISABLE ROW LEVEL SECURITY;
NOTIFY pgrst, 'reload schema';
```

`apply-all-migrations.sql` 마지막에도 동일한 RLS 비활성화 구문이 포함되어 있습니다.

## 1단계: 로컬에서 코드 수정

Cursor에서 수정 후 저장합니다.

로컬 개발 서버:

```bash
npm install
npm run dev
```

- 기본 주소: http://localhost:3000  
- 로컬 환경 변수: 프로젝트 루트 `.env.local` (Git에 올리지 않음)

---

## 2단계: GitHub에 푸시

Cursor **터미널**에서:

```bash
git add .
git commit -m "변경 내용 설명"
git push origin master
```

- `main` 대신 기본 브랜치가 `master`라면 `git push origin master` 사용  
- 커밋 메시지는 실제 변경 내용으로 바꿔서 작성  
- `.env.local`, `.next`는 **커밋하지 않습니다**

---

## 3단계: Replit에서 동기화 (매번 배포 전 실행)

Replit 왼쪽 **Tools** → **Shell** 을 연 뒤, 아래를 **순서대로** 실행합니다.

### 3-1. 최신 코드 가져오기 (필수)

```bash
git fetch origin main
git reset --hard origin/main
```

- 브랜치가 `master`면 `main` 대신 `master` 로 바꿉니다.

### 3-2. 패키지 설치 (필수)

동기화 후 **항상** 실행하는 것을 권장합니다:

```bash
npm install
```

### 3-3. 프로덕션 빌드 (필수)

Next.js는 배포 전 빌드가 필요합니다:

```bash
npm run build
```

빌드가 성공해야 Republish 후 정상 동작합니다.

### 3-4. 한 번에 실행하려면

아래를 복사해 Shell에 붙여 넣고 실행:

```bash
git fetch origin master && git reset --hard origin/master && npm install && npm run build
```

## 4단계: 재배포

1. Replit 오른쪽 상단 **Deploy** 영역에서 **Republish** (또는 **Deploy**) 클릭  
2. 배포가 끝난 뒤 제공되는 URL로 접속해 동작 확인  
3. `/login` → 로그인 → `/dashboard`, `/dashboard/announcements` 화면 확인

## Replit Secrets 설정

Replit **Tools** → **Secrets** 에 다음이 설정되어 있는지 확인하세요.

### 필수

| Secret 이름 | 설명 |
|-------------|------|
| `NEXT_PUBLIC_SUPABASE_URL` | Supabase 프로젝트 URL (`https://xxxx.supabase.co`) |
| `SUPABASE_SERVICE_ROLE_KEY` | Service Role Key (Dashboard → Settings → API) |
| `AUTH_SECRET` | 세션 쿠키 서명용 랜덤 문자열 (`openssl rand -base64 32` 등으로 생성) |

### 선택 (네트워크·기능별)

| Secret 이름 | 설명 |
|-------------|------|
| `SUPABASE_SSL_VERIFY` | 회사망 TLS 오류 시 `0` — 개발·내부망 한정 |
| `ANTHROPIC_API_KEY` | 발주요약 AI 엔진(클로드) 사용 시 |
| `ANTHROPIC_MODEL` | (선택) 클로드 모델명. 기본 `claude-sonnet-4-6` |
| `CLAUDE_TLS_INSECURE` | (선택) 클로드 API TLS 오류 시 `true` |
| `FRIENDLI_TOKEN` | 발주요약 AI 엔진(LG엑사원) 사용 시 |
| `FRIENDLI_MODEL` | (선택) 엑사원 모델명 |
| `FRIENDLI_SSL_VERIFY` | (선택) 엑사원 API TLS 오류 시 `0` |
| `HWP_CONVERT_PYTHON` | (선택) HWP 텍스트 추출용 Python 경로. Replit/Linux 기본 `python3` |

이 값들은 **GitHub에 올리지 말고**, Replit Secrets 에만 넣습니다.

### 발주요약 HWP 처리 (Replit 필수)

HWP/HWPX 첨부파일 요약에는 **Python 3 + hwpkit** 이 필요합니다. Replit 배포 환경에는 Node.js만 있으면 `spawn python3 ENOENT` 오류가 납니다.

저장소의 `.replit` 에 `python3` 모듈이 포함되어 있고, `npm run build` 시 `hwpkit`·`lxml` 이 자동 설치됩니다. **최초 1회** 아래를 Replit Shell에서 실행한 뒤 Republish 하세요.

```bash
git fetch origin master && git reset --hard origin/master && npm install && npm run build
```

빌드 로그에 `[install-hwp-python-deps] python3 -m pip install ...` 가 보이면 정상입니다.

동작 확인 (Shell):

```bash
python3 --version
python3 -c "import hwpkit; print('hwpkit OK')"
```

문제가 계속되면 Replit **Secrets** 에 `HWP_CONVERT_PYTHON=python3` 를 추가한 뒤 Republish 합니다.

## Replit 실행 설정 (최초 1회)

저장소에 `.replit` 파일이 포함되어 있습니다. Replit Deployment 설정에서 아래와 유사한지 확인하세요.

| 항목 | 권장 값 |
|------|---------|
| Build | `npm run build` |
| Run | `npm run start` |

`npm run start`는 `scripts/start-production.mjs`를 실행하며, Replit이 제공하는 `PORT` 환경 변수에 `0.0.0.0`으로 바인딩합니다.

## 반영이 안 될 때 체크리스트

| 확인 항목 | 조치 |
|-----------|------|
| Replit에서 `git status` 로 브랜치/파일 상태 확인 | `git fetch origin main && git reset --hard origin/main` 다시 실행 |
| 동기화 후 `npm install` / `npm run build` 안 함 | 3단계 전체 실행 후 Republish |
| Republish만 하고 Shell에서 동기화 안 함 | **3단계 전체** 실행 후 다시 Republish |
| Replit이 예전 커밋을 가리킴 | GitHub 푸시 성공 여부 확인 후, Replit Shell에서 `git log -1` 로 최신 커밋인지 확인 |
| 환경 변수 미설정 | Replit Secrets 3종(`NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `AUTH_SECRET`) 확인 후 Republish |
| `Could not find the table 'public.bid_users'` | Supabase SQL Editor에서 `014_bid_users.sql` 또는 `apply-all-migrations.sql` 실행 |
| `Could not find the table 'public.khnp_bid_notice'` | Supabase SQL Editor에서 `apply-all-migrations.sql` 실행 |
| `new row violates row-level security policy` | 위 **RLS 오류 예방** SQL 실행 |
| Supabase 연결 실패 (fetch failed) | Secrets에 `SUPABASE_SSL_VERIFY=0` 추가 후 Republish (내부망 한정) |
| 빌드 실패 | Replit Shell에서 `npm run build` 로 오류 메시지 확인 후 수정·푸시 |
| 입찰공고 목록이 비어 있음 | `crawl_sites`·`khnp_bid_notice` 데이터 존재 여부 확인 |
| 발주요약 실패 | `ANTHROPIC_API_KEY` 또는 `FRIENDLI_TOKEN` Secrets 설정 확인 |
| 발주요약 HWP 오류 (`spawn python3 ENOENT`) | 위 **발주요약 HWP 처리** 절차 실행 후 Republish. `python3 -c "import hwpkit"` 로 확인 |

## Supabase 연결 참고

- 이 앱은 **Supabase Auth를 사용하지 않습니다.** 사용자는 Supabase `bid_users` + bcrypt 세션 쿠키로 인증합니다.
- DB의 `user_id` 컬럼에는 `bid_users.id` 문자열이 저장됩니다.
- Service Role Key는 **서버에서만** 사용합니다. 브라우저·GitHub에 노출하지 마세요.
- IT 일정관리 시스템(`it_*`)과 테이블이 겹치지 않도록 `khnp_bid_*`, `user_bid_*` 접두사를 사용합니다. **동일 Supabase 프로젝트를 공유해도 됩니다.**

## 요약: Replit에서 배포할 때마다

1. Shell 열기  
2. `git fetch origin main && git reset --hard origin/main && npm install && npm run build` 실행  
3. Republish 클릭  
4. 배포 URL에서 `/login` 동작 확인  

이 순서를 지키면 로컬에서 푸시한 내용이 Replit 배포에 반영됩니다.
